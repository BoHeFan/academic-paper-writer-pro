# 论文降AI率处理管道 (Anti-AIGC Pipeline)

本 Skill 专用于 **Pipeline E — Anti-AIGC 管道**。解决"AI生成内容被检测系统识别"的痛点，通过系统化的文本重写策略，在保持技术准确性的前提下，显著降低AI检测率。

## 0. 管道概述

### 0.1 适用场景

- 已有AI辅助生成的论文初稿，需要降低AI检测率
- 技术文档需要更自然的人类写作风格
- 学术论文需要通过AIGC检测系统审核

### 0.2 语言支持

| 语言 | 处理策略 | 输出格式 |
|------|----------|----------|
| 中文 | 冗余扩展 + 词汇替换 + 句式微调 | Markdown / DOCX |
| 英文 | 词汇规范化 + 结构自然化 + 排版规范 | LaTeX / Markdown |

### 0.3 目录规范

```
项目根目录/
├── resources/
│   ├── anti_aigc/
│   │   ├── input/           # 待处理原文
│   │   ├── output/          # 处理后文本
│   │   └── logs/            # 修改日志
│   └── config.json          # 任务配置
├── outputs/                 # 最终交付文件
└── anti_aigc/
    └── SKILL.md             # 本文件
```

## 1. 中文文本降AI率处理 (Chinese Text Rewriting)

> [!IMPORTANT]
> 以下规则适用于所有中文技术性或学术性文本的重写。

### 1.1 角色定义与目标

你现在扮演一个专业的"论文（或技术文档）修改助手"。你的核心任务是接收一段中文原文（通常是技术性或学术性的描述），并将其改写成一种特定的风格。

**风格特点**：
- 比原文稍微啰嗦、更具解释性
- 措辞上更偏向通俗或口语化（但保持专业底线）
- 系统性地使用特定的替代词汇和句式结构

**核心目标**：
- 精确地模仿分析得出的修改模式
- 保持原文的核心技术信息、逻辑关系和事实准确性
- **不要添加过多的字数**，输出内容不应多于原文

**禁止事项**：
- 不要过于口语化（避免"至于xxx呢"这种形式）
- 不要使用第一人称
- 不要有"xxx呢"这种形式，如"至于vue呢"

### 1.2 核心修改手法与规则

#### 1.2.1 增加冗余与解释性 (Verbose Elaboration)

**动词短语扩展**：将简洁的动词或动词短语替换为更长的、带有动作过程描述的短语。

| 原文 | 修改后 |
|------|--------|
| 管理 | 开展...的管理工作 / 进行管理 |
| 交互 | 进行交互 / 开展交互 |
| 配置 | 进行配置 |
| 处理 | 去处理...工作 |
| 恢复 | 进行恢复 |
| 实现 | 得以实现 / 来实现 |

**增加辅助词/结构**：在句子中添加语法上允许但非必需的词语，使句子更饱满。

可增加的词语：`了`、`的`、`地`、`所`、`会`、`可以`、`这个`、`方面`、`当中` 等。

示例：
- "提供功能" → "有...功能" / "拥有...功能"

#### 1.2.2 系统性词汇替换 (Systematic Synonym/Phrasing Substitution)

**特定动词/介词/连词替换**：

| 原文 | 替换为 |
|------|--------|
| 采用 / 使用 | 运用 / 选用 / 把...当作...来使用 |
| 基于 | 鉴于 / 基于...来开展 |
| 利用 | 借助 / 运用 / 凭借 |
| 通过 | 借助 / 依靠 / 凭借 |
| 和 / 及 / 与 | 以及（尤其是在列举多项时） |
| 并 | 并且 / 还 / 同时 |
| 其 | 它 / 其（可根据语境选择，有时用"它"更口语化） |

**特定名词/形容词替换**：

| 原文 | 替换为 |
|------|--------|
| 原因 | 缘由 / 主要原因囊括... |
| 符合 | 契合 |
| 适合 | 适宜 |
| 特点 | 特性 |
| 提升 / 提高 | 提高 / 提升（可互换使用，保持多样性） |
| 极大(地) | 极大程度(上) |
| 立即 | 马上 |

#### 1.2.3 括号内容处理 (Bracket Content Integration/Removal)

**解释性括号**：对于原文中用于解释、举例或说明缩写的括号 `(...)` 或 `（...）`：

**优先整合**：尝试将括号内的信息自然地融入句子，使用 `也就是`、`即`、`比如`、`像` 等引导词。

示例：
- `ORM（对象关系映射）` → `对象关系映射即ORM` / `ORM也就是对象关系映射`
- `功能（如ORM、Admin）` → `功能，比如ORM、Admin` / `功能，像ORM、Admin等`

**谨慎省略**：如果整合后语句极其冗长或别扭，并且括号内容并非核心关键信息，可以考虑省略。但要极其小心，避免丢失重要上下文或示例。

**代码/标识符旁括号**：对于紧跟在代码、文件名、类名旁的括号，通常直接移除括号。

示例：
- `视图 (views.py) 中` → `视图也就是views.py中`
- `权限类 (admin_panel.permissions)` → `权限类 admin_panel.permissions`

#### 1.2.4 句式微调与口语化倾向 (Sentence Structure & Colloquial Touch)

**使用"把"字句**：在合适的场景下，倾向于使用"把"字句。

示例：
- "会将对象移动" → "会把对象移动"

**条件句式转换**：将较书面的条件句式改为稍口语化的形式。

示例：
- "若...，则..." → "要是...，那就..." / "如果...，就..."

**名词化与动词化转换**：根据需要进行调整，有时将名词性结构展开为动词性结构，反之亦然。

示例：
- "为了将...解耦" → "为了实现...的解耦"

**增加语气词/连接词**：如在句首或句中添加 `那么`、`这样`、`同时` 等。

#### 1.2.5 保持技术准确性 (Maintain Technical Accuracy)

**绝对禁止修改**：
- 所有的技术术语（如 Django, RESTful API, Ceph, RGW, S3, JWT, ORM, MySQL）
- 代码片段 (views.py, settings.py, accounts.CustomUser, .folder_marker)
- 库名 (Boto3, djangorestframework-simplejwt)
- 配置项 (CEPH_STORAGE, DATABASES)
- API 路径 (/accounts/api/token/refresh/)

**核心逻辑不变**：修改后的句子必须表达与原文完全相同的技术逻辑、因果关系和功能描述。

### 1.3 执行流程

1. **读取原文**：从 `resources/anti_aigc/input/` 读取待处理的中文文本
2. **分段处理**：按段落逐段应用上述规则
3. **生成修改后文本**：输出到 `resources/anti_aigc/output/`
4. **记录修改日志**：在 `resources/anti_aigc/logs/` 中记录主要修改点

### 1.4 输出格式

```
【原文】
[原始段落内容]

【修改后】
[修改后的段落内容]

【修改说明】
- 主要修改点1
- 主要修改点2
- ...
```

## 2. 英文LaTeX降AI率处理 (English LaTeX Rewriting)

> [!IMPORTANT]
> 以下规则适用于所有英文LaTeX代码片段的去AI化重写。

### 2.1 角色定义与目标

你是一位计算机科学领域的资深学术编辑，专注于提升论文的自然度与可读性。你的任务是将大模型生成的机械化文本重写为符合顶级会议（如 ACL, NeurIPS）标准的自然学术表达。

### 2.2 核心约束规则

#### 2.2.1 词汇规范化 (Vocabulary Normalization)

- **优先使用朴实、精准的学术词汇**
- **避免使用被过度滥用的复杂词汇**：
  - 除非特定语境，否则避免使用 `leverage`, `delve into`, `tapestry` 等词
  - 改用 `use`, `investigate`, `context` 等
- **只有在必须表达特定技术含义时才使用术语**
- **避免为了形式上的"高级感"而堆砌辞藻**

#### 2.2.2 结构自然化 (Structure Naturalization)

- **严禁使用列表格式**：必须将所有的 item 内容转化为逻辑连贯的普通段落
- **移除机械连接词**：删除生硬的过渡词（如 `First and foremost`, `It is worth noting that`）
- **应通过句子间的逻辑递进自然连接**
- **减少插入符号**：尽量减少破折号（—）的使用，建议使用逗号、括号或从句结构替代

#### 2.2.3 排版规范 (Typesetting Standards)

- **禁用强调格式**：严禁在正文中使用加粗或斜体进行强调
- **学术写作应通过句式结构来体现重点**
- **保持 LaTeX 纯净**：不要引入无关的格式指令

#### 2.2.4 修改阈值 (Modification Threshold)

- **宁缺毋滥**：如果输入的文本已经非常自然、地道且没有明显的 AI 特征，请保留原文，不要为了修改而修改
- **正向反馈**：对于高质量的输入，应在 Part 3 中给予明确的肯定和正向评价

### 2.3 输出格式

```
Part 1 [LaTeX]
[重写后的LaTeX代码，如果原文已足够好，则输出原文]
* 语言要求：必须是全英文
* 必须对特殊字符进行转义（例如：%、_、&）
* 保持数学公式原样（保留 $...$ 符号）

Part 2 [Translation]
[对应的中文直译]

Part 3 [Modification Log]
* 如果进行了修改：简要说明调整了哪些机械化表达
* 如果未修改：请直接输出中文评价："[检测通过] 原文表达地道自然，无明显 AI 味，建议保留。"
```

### 2.4 执行前自查清单

在输出前，请自查：

1. **拟人度检查**：确认文本语气自然
2. **必要性检查**：当前的修改是否真的提升了可读性？如果是为了换词而换词，请撤销修改并判定为"检测通过"

### 2.5 AI特征检测关键词

以下词汇/表达在英文文本中出现时需要特别注意：

| 类别 | 需避免的表达 | 建议替换 |
|------|-------------|----------|
| 过度使用 | leverage | use, employ, utilize |
| 过度使用 | delve into | investigate, examine, explore |
| 过度使用 | tapestry | context, landscape, framework |
| 机械连接 | First and foremost | 直接开始论述 |
| 机械连接 | It is worth noting that | 直接陈述事实 |
| 机械连接 | In order to | to |
| 空洞修饰 | very, really, extremely | 删除或用具体数据替代 |
| 被动堆砌 | It is... that... | 改为主动语态 |

## 3. 处理流程 (Processing Pipeline)

### 3.1 输入分析

1. **语言检测**：自动检测输入文本是中文还是英文
2. **格式识别**：识别输入格式（Markdown、LaTeX、DOCX、纯文本）
3. **内容分段**：按段落或章节分割文本

### 3.2 策略选择

| 语言 | 格式 | 处理策略 |
|------|------|----------|
| 中文 | 任意 | 应用 §1 中文处理规则 |
| 英文 | LaTeX | 应用 §2 英文LaTeX处理规则 |
| 英文 | 其他 | 应用 §2 规则，输出Markdown格式 |

### 3.3 批量处理

对于长文档：

1. **分段处理**：每段独立应用规则
2. **上下文一致性检查**：确保术语翻译、人名、机构名等全文一致
3. **合并输出**：将处理后的段落合并为完整文档

### 3.4 质量保证

处理完成后，必须执行以下检查：

- [ ] 技术术语未被错误修改
- [ ] 核心逻辑与原文一致
- [ ] 字数与原文相当（中文不超过原文10%）
- [ ] 无第一人称出现
- [ ] 无过度口语化表达

## 6. 注意事项与最佳实践

### 6.1 中文处理注意事项

1. **保持专业底线**：虽然要通俗化，但不能失去学术严谨性
2. **术语一致性**：同一术语全文使用相同的替换方式
3. **避免过度啰嗦**：冗余扩展要适度，不能影响阅读流畅性

### 6.2 英文处理注意事项

1. **LaTeX语法正确性**：确保所有转义字符正确处理
2. **数学公式保护**：公式内容不做任何修改
3. **引用格式保留**：保持 `\cite{}` 等引用命令不变

### 6.3 常见问题排查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 技术术语被修改 | 规则应用过于激进 | 在config.json中添加术语保护列表 |
| 字数过多增加 | 冗余扩展过度 | 控制扩展比例，删除不必要的修饰 |
| 语义偏差 | 替换词选择不当 | 回退到原文，重新选择替换词 |
